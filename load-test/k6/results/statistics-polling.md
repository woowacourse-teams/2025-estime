# 시나리오 3: 통계 조회 폴링 (statistics-polling)

## 목적
가장 빈번하게 호출되는 통계 조회 API의 읽기 성능과 캐시 효과 측정.
50개 방에 분산된 캐시 키 환경에서 현실적인 부하 확인.

## 테스트 설계

**setup**: 방 50개 생성, 각 방에 참여자 10명 등록 + 투표 완료 (총 500명, 7일 x 48슬롯 = 336개 datetime)
**본 테스트**: 50개 방 중 랜덤 선택하여 통계 조회 반복 (sleep 0.5s)

- Caffeine 캐시 TTL 500ms → 초당 캐시 키당 최대 2회 미스
- 50개 캐시 키 → 초당 최대 100회 캐시 미스 (DB 쿼리 발생)
- VU당 1 iteration = API 호출 1회 (통계 조회)
- VU당 1 iteration 소요 시간 ~ 0.55초

## 1~2차: 150 VU (안정성 확인)

| 구간 | VU |
|------|-----|
| Ramp-up | 0->20, 30s |
| Push | 20->50, 1m |
| Stress | 50->100, 1m |
| Max | 100->150, 1m |
| Cool-down | 150->0, 30s |

| 지표 | 1차 | 2차 |
|------|-----|-----|
| 총 요청 | 29,775 | 29,455 |
| RPS | 111/s | **110/s** |
| p50 | 44ms | **43ms** |
| p95 | 224ms | **274ms** |
| p99 | 419ms | **580ms** |
| max | 1.5s | 1.54s |
| 에러율 | **0%** | **0%** |
| 통계 조회 | 28,725회 | 28,405회 |
| 소요 시간 | 4분 28초 | 4분 28초 |

### 분석
- 150 VU에서도 **에러율 0%, p50 = 43~44ms**로 안정적
- 읽기 전용이므로 쓰기(시나리오 1, 2) 대비 월등히 높은 VU 수용 가능
- p99 = 419~580ms 범위 → 캐시 미스 + GC STW 복합 영향
- Caffeine 캐시가 효과적으로 작동하여 DB 부하를 흡수
- 2회 실행 결과 일관성 확인 (RPS, p50 거의 동일)
- **결론: 150 VU까지 안정. 3차에서 더 높은 VU로 한계 탐색**

## 3차: 500 VU (한계 탐색)

| 구간 | VU |
|------|-----|
| Fast ramp | 0->150, 30s |
| Sustain | 150, 1m |
| Push | 150->300, 30s |
| Sustain | 300, 1m |
| Max hunt | 300->500, 30s |
| Sustain | 500, 1m |
| Cool-down | 500->0, 30s |

| 지표 | 결과 |
|------|------|
| 총 요청 | 56,487 |
| RPS | **173/s** |
| p50 | **1.01s** (임계 초과) |
| p95 | **2.29s** (임계 초과) |
| p99 | **2.81s** |
| max | 3.56s |
| 에러율 | **0%** |
| 통계 조회 | 55,437회 |
| 소요 시간 | 5분 27초 |

### 분석
- **에러율 0%**이지만 p50이 1.01s로 임계(100ms) 대폭 초과
- 150VU→500VU에서 p50이 43ms→1.01s로 약 23배 증가
- **Tomcat 스레드 풀(15개) 포화**: 500 VU가 15개 스레드를 공유하며 대기열 발생
- 캐시 히트에도 불구하고 스레드 대기 시간이 지배적
- RPS는 150VU(110/s) → 500VU(173/s)로 약 57% 증가에 그침 → 스레드 병목으로 처리량 한계
- **읽기 한계 지점: 150~300 VU 사이에서 p50 100ms 경계 (RPS 110~173)**

## 4차: 1500 VU (극한 탐색) — 에러 발생으로 자동 중단

| 구간 | VU |
|------|-----|
| Fast ramp | 0->500, 30s |
| Sustain | 500, 1m |
| Push | 500->750, 30s |
| Sustain | 750, 1m |
| Extreme | 750->1000, 30s |
| Sustain | 1000, 1m |
| Breaking | 1000->1500, 30s |
| Sustain | 1500, 1m |
| Cool-down | 1500->0, 30s |

| 지표 | 결과 |
|------|------|
| 총 요청 | 21,965 |
| RPS | **151/s** |
| p50 | **2.22s** |
| p95 | **3.16s** |
| p99 | **3.34s** |
| max | 4.04s |
| 에러율 | **0.01%** (4건) |
| 최대 도달 VU | **~733** (1500 목표 중) |
| 소요 시간 | 2분 26초 (자동 중단) |

### 에러 상세
```
connection reset by peer (read tcp → 3.39.35.178:443)
```
~733 VU 시점에서 서버가 TCP 연결을 강제로 끊음. 4건 동시 발생.

### 분석
- **733 VU에서 처음으로 에러 발생** — 전체 부하테스트 중 유일한 에러 지점
- 에러 원인: `connection reset by peer` → 서버(또는 앞단 인프라)가 더 이상 연결을 수용할 수 없어 TCP RST 전송
- Tomcat accept-count(5) + max-threads(15) = 동시 처리 20개까지. 나머지 713개 요청이 OS 레벨에서 대기하다 일부 타임아웃/거부
- RPS가 173(500VU) → 151(733VU)로 오히려 **감소** → 서버가 과부하로 처리 효율 저하
- p99=3.34s로 3차(2.81s) 대비 소폭 증가 — 스레드 대기열이 이미 포화되어 큰 차이 없음
- **결론: 읽기 작업의 절대 한계는 ~730 VU. 이 이상에서 서버가 연결을 거부하기 시작함**
