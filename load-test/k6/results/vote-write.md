# 시나리오 2: 투표 쓰기 (vote-write)

## 목적
동시 투표 수정 쓰기 작업의 안정성 확인.
각 VU = 1명의 실제 사용자가 투표를 수정하는 현실적 시나리오.

## 테스트 설계

**setup**: 방 50개 생성, 각 방에 참여자 10명 등록 (총 500명), 7일 x 48슬롯 = 336개 datetime 조합
**본 테스트**: 각 VU가 고유한 방/참여자로 반복 투표 수정 (~60% 슬롯 랜덤 선택, sleep 0.5s)

- VU ID 기반으로 방/참여자 고정 할당 (동시 수정 충돌 방지)
- VU당 1 iteration = API 호출 1회 (투표 수정)
- VU당 1 iteration 소요 시간 ~ 0.55초

### 설계 과정에서의 발견

초기 버전에서는 랜덤 방/참여자 선택 방식이었으나, **13 VU에서 1건의 서버 에러 발생**.
원인: 두 VU가 **같은 참여자**를 동시에 선택하여 투표 수정 시 race condition 발생.

```
VU A: SELECT 참여자X 투표 → [슬롯1, 슬롯2]
VU B: SELECT 참여자X 투표 → [슬롯1, 슬롯2]  (동일 상태 읽기)
VU A: DELETE 슬롯1, INSERT 슬롯3 → Commit ✓
VU B: DELETE 슬롯1 (이미 삭제!) → 서버 에러 💥
```

서버 코드에 비관적 잠금이 없어 동시 수정 시 DB 제약조건 위반 발생.
실제 서비스에서는 같은 사용자가 동시에 투표하는 일이 없으므로, VU별 고유 참여자 할당으로 수정.

## 1~2차: 50 VU (안정성 확인)

| 구간 | VU |
|------|-----|
| Ramp-up | 0->10, 30s |
| Sustain | 10->20, 1m |
| Push | 20->30, 1m |
| Stress | 30->50, 1m |
| Cool-down | 50->0, 30s |

| 지표 | 1차 | 2차 |
|------|-----|-----|
| 총 요청 | 10,789 | 10,774 |
| RPS | 43/s | **43/s** |
| p50 | 40ms | **39ms** |
| p95 | 66ms | **62ms** |
| p99 | 196ms | **134ms** |
| max | 867ms | 1.86s |
| 에러율 | **0%** | **0%** |
| 투표 수정 | 10,239회 | 10,224회 |
| 소요 시간 | 4분 10초 | 4분 10초 |

### 분석
- 50 VU 동시 투표 쓰기에서 **에러율 0%, p95 = 62~66ms**로 매우 안정적
- 2회 실행 결과가 거의 동일하여 **재현성 확인**
- 투표 수정은 DELETE + INSERT 조합이지만 각 참여자가 독립적이므로 DB 경합 최소
- max 1.86s는 GC STW로 추정 (VU 수와 무관하게 간헐 발생)
- **결론: 50 VU까지 투표 쓰기 안정. 3차에서 한계 탐색 진행**
