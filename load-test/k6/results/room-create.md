# 시나리오 1: 방 생성 + 참여자 등록 (room-create)

## 목적
방 생성과 참여자 등록 쓰기 작업의 안정성 확인.
이후 시나리오의 setup이 안전하다는 확신을 얻기 위한 선행 테스트.

## 테스트 설계

**setup**: 슬롯 데이터만 미리 계산 (7일 x 48슬롯 = 336개 datetime 조합)
**본 테스트**: 각 VU가 매 iteration마다 방 1개 생성 + 참여자 10명 등록 (sleep 1s)

- VU당 1 iteration = API 호출 11회 (방 생성 1 + 참여자 등록 10)
- VU당 1 iteration 소요 시간 ~ 1.4초

## 1~2차: 30 VU (안정성 확인)

| 구간 | VU |
|------|-----|
| Warm-up | 0->5, 30s |
| Ramp | 5->10, 1m |
| Sustain | 10->20, 1m |
| Stress | 20->30, 1m |
| Cool-down | 30->0, 30s |

| 지표 | 1차 | 2차 |
|------|-----|-----|
| 총 요청 | 25,828 | 25,839 |
| RPS | 107/s | **107/s** |
| p50 | 30ms | **31ms** |
| p95 | 78ms | **80ms** |
| p99 | 118ms | **128ms** |
| max | 1.86s | 1.06s |
| 에러율 | **0%** | **0%** |
| 방 생성 | 2,348개 | 2,349개 |
| 참여자 등록 | ~23,480명 | ~23,490명 |
| 소요 시간 | 4분 | 4분 |

### 분석
- 30 VU 동시 쓰기에서 **에러율 0%, p99 = 118~128ms**로 매우 안정적
- 2회 실행 결과가 거의 동일하여 **재현성 확인**
- **결론: 이후 시나리오의 setup(방 50개 + 참여자 500명 순차 생성)은 안전하다고 확신**

## 3차: 100 VU (한계 탐색)

| 구간 | VU |
|------|-----|
| Fast ramp | 0->30, 30s |
| Sustain | 30, 1m |
| Push | 30->60, 30s |
| Sustain | 60, 1m |
| Max hunt | 60->100, 30s |
| Sustain | 100, 1m |
| Cool-down | 100->0, 30s |

| 지표 | 결과 |
|------|------|
| 총 요청 | 77,220 |
| RPS | **257/s** |
| p50 | **115ms** (임계 초과) |
| p95 | **290ms** |
| p99 | **516ms** |
| max | 1.69s |
| 에러율 | **0%** |
| 방 생성 | 7,020개 |
| 참여자 등록 | ~70,200명 |
| 소요 시간 | 5분 |

### 분석
- **에러율 0%**이지만 p50이 115ms로 임계(100ms) 초과
- 30VU->100VU에서 p50이 30ms->115ms로 약 4배 증가 -> **Hikari 커넥션 풀(5개) 포화로 대기 발생**
- 에러 없이 느려지는 구간 (graceful degradation)
- **쓰기 한계 지점: 60~100 VU (RPS 170~257) 사이에서 p50 100ms 경계**
