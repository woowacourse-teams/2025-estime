# 아인슈타임(Estime) 부하테스트 결과 보고서

## 1. 개요

아인슈타임 프로덕션 서버를 대상으로 k6 부하테스트를 수행하여 서비스의 성능 한계와 병목 지점을 파악하였다.
모든 시나리오는 **에러율 0%** 기준으로 설계했으며, 안정성 확인(1~2차) 후 한계 탐색(3차)을 진행하였다.

### 테스트 환경

| 항목 | 값 |
|------|-----|
| 대상 서버 | 프로덕션 (`prod.estime.today`) |
| Tomcat max-threads | **15** (accept-count: 5) |
| Hikari max-pool-size | **5** (connection-timeout: 30s) |
| Caffeine 캐시 TTL | **500ms** |
| 테스트 도구 | Grafana k6 (로컬 실행) |
| 테스트 데이터 | 최악 케이스 — 7일 x 48슬롯 = **336 datetime/방** |
| Nginx 레이트리밋 | 테스트 기간 동안 비활성화 |

### 테스트 방법론

각 시나리오마다 **1~2차**(안정성 확인)와 **3차**(한계 탐색)로 나누어 진행하였다.

- **1~2차**: 적절한 부하에서 에러 없이 버티는지 확인. 2회 실행으로 **재현성 검증**.
- **3차**: VU를 점진적으로 높여 p50이 100ms를 넘는 **성능 경계 지점**을 탐색.
- 모든 시나리오에서 `http_req_failed: rate==0` (에러 시 즉시 중단) 임계값 적용.

## 3. 시나리오별 상세 분석

### 시나리오 1: 방 생성 + 참여자 등록

**구성**: 각 VU가 방 1개 생성 + 참여자 10명 등록 = iteration당 API 11회 호출 (sleep 1s).

#### 안정 구간 (30 VU)

p50=30ms, p95=78ms로 모든 요청이 100ms 이내에 처리되었다.
Hikari 커넥션 5개가 30 VU의 DB 요청을 충분히 소화한다.
각 쓰기 쿼리가 단순 INSERT이므로 커넥션 점유 시간이 짧아 **풀 회전율이 높다**.
2회 실행 결과가 거의 동일하여 재현성을 확인하였다.

#### 한계 구간 (100 VU)

| 지표 | 30 VU | 100 VU | 증가율 |
|------|-------|--------|--------|
| p50 | 30ms | 115ms | **3.8x** |
| p95 | 78ms | 290ms | **3.7x** |
| p99 | 118ms | 516ms | **4.4x** |
| RPS | 107/s | 257/s | 2.4x |

p50이 30ms→115ms로 약 4배 증가하였다. 100 VU가 동시에 DB 커넥션을 요청하면 5개만 즉시 처리되고 95개는 대기한다.
p99의 증가율(4.4x)이 p50(3.8x)보다 큰 것은, 대기열 뒷쪽 요청일수록 누적 대기 시간이 길어지기 때문이다.

VU가 3.3배 늘었는데 RPS는 2.4배 증가에 그친 것은 **Hikari 커넥션 풀 포화로 처리량 증가가 둔화**되기 시작했다는 의미이다.

**성능 경계: 60~100 VU (RPS 170~257) 사이에서 p50이 100ms를 넘는다.**

#### 극한 구간 (4차, 500 VU)

| 지표 | 100 VU | 500 VU | 증가율 |
|------|--------|--------|--------|
| p50 | 115ms | 615ms | **5.3x** |
| p95 | 290ms | 1.48s | **5.1x** |
| p99 | 516ms | 1.89s | **3.7x** |
| RPS | 257/s | 335/s | 1.3x |

500 VU에서도 에러율 0%. RPS가 257→335로 VU 5배 대비 30% 증가에 그침 → Hikari 풀 처리량 포화.
iteration당 11회 API 호출이므로 각 호출에 615ms가 걸리면 1 iteration에 약 8.5초가 소요된다.
Hikari connection-timeout(30s)에는 여전히 여유가 있어 에러로 이어지지 않는다.

### 시나리오 2: 투표 쓰기

**구성**: 50개 방 x 10명 참여자를 setup으로 생성 후, 각 VU가 고유한 참여자로 반복 투표 수정.
1회 투표 수정 = SELECT(기존 투표) + DELETE(제거 대상) + INSERT(추가 대상) + 캐시 evict + SSE 이벤트 발행.
336개 슬롯 중 ~60%(약 200개)를 매 iteration마다 랜덤 선택한다 (sleep 0.5s).

#### 안정 구간 (50 VU)

p50=39ms, p95=62ms로 매우 안정적이다.
방 생성(p50=30ms)보다 약간 느린 것은 SELECT→DELETE→INSERT 3단계 트랜잭션 때문이다.
p95=62ms가 시나리오 1(78ms)보다 오히려 낮은 이유는, iteration당 API 호출이 1회(투표 수정만)라서 VU 간 경합 빈도가 낮기 때문이다.

#### 한계 구간 (150 VU)

| 지표 | 50 VU | 150 VU | 증가율 |
|------|-------|--------|--------|
| p50 | 39ms | 359ms | **9.2x** |
| p95 | 62ms | 855ms | **13.8x** |
| p99 | 134ms | 1.06s | **7.9x** |
| RPS | 43/s | 98/s | 2.3x |

p50이 39ms→359ms로 **9배 폭증**하였다. 시나리오 1의 4배보다 훨씬 급격한 이유는 **트랜잭션당 DB 커넥션 점유 시간**의 차이에 있다:

- 방 생성: INSERT 1회 → 커넥션을 빠르게 반환
- 투표 수정: SELECT + batch DELETE + batch INSERT → 커넥션을 **3배 이상 오래 점유**

커넥션 1개의 점유 시간이 길어지면 풀 회전율이 떨어진다.
5개 커넥션의 실질 처리 용량이 줄어들면서 같은 VU 수에서도 대기 시간이 훨씬 길어진다.
p95=855ms는 **커넥션 대기 ~650ms + 실제 쿼리 실행 ~200ms**의 조합으로 추정된다.

VU 3배 증가에 RPS 2.3배 증가로, 시나리오 1과 동일한 **Hikari 풀 포화 패턴**이 나타났다.

**성능 경계: 50~100 VU (RPS 43~98) 사이에서 p50이 100ms를 넘는다.**

#### 극한 구간 (4차, 500 VU)

| 지표 | 150 VU | 500 VU | 증가율 |
|------|--------|--------|--------|
| p50 | 359ms | 2.1s | **5.8x** |
| p95 | 855ms | 4.22s | **4.9x** |
| p99 | 1.06s | 5.78s | **5.5x** |
| RPS | 98/s | 104/s | **1.06x (거의 변화 없음)** |

500 VU에서도 에러율 0%. 그러나 p50=2.1s로 요청의 절반이 2초 이상 걸린다.
가장 주목할 점은 **RPS가 98→104로 거의 증가하지 않은 것**이다.
VU를 3.3배 올려도 처리량이 6% 증가에 그친 것은, Hikari 커넥션 5개의 물리적 처리 한계(~100 RPS)에 도달했다는 의미이다.
투표 수정은 커넥션 점유 시간이 길어 커넥션당 초당 ~20회 처리가 한계이고, 5개 x 20 = 100 RPS가 이론적 최대값이 된다.

#### 설계 과정에서 발견한 Race Condition

초기 테스트에서 랜덤 참여자 선택 방식을 사용했을 때, 13 VU에서 서버 에러가 1건 발생하였다.

```
VU A: SELECT 참여자X 투표 → [슬롯1, 슬롯2]
VU B: SELECT 참여자X 투표 → [슬롯1, 슬롯2]  (동일 상태 읽기)
VU A: DELETE 슬롯1, INSERT 슬롯3 → Commit 성공
VU B: DELETE 슬롯1 (이미 삭제됨!) → 서버 에러
```

서버의 투표 수정 로직(`CompactRoomApplicationService.updateParticipantVotes()`)이 `SELECT → subtract → DELETE → INSERT` 패턴으로 구현되어 있으나, **비관적 잠금(Pessimistic Lock)이 없다**.
동일 참여자에 대한 동시 요청이 들어오면 한쪽이 이미 삭제된 레코드를 다시 삭제하려 하여 예외가 발생한다.

실제 서비스에서는 같은 사용자가 동시에 두 번 투표하는 상황이 사실상 없으므로, 테스트를 VU별 고유 참여자 할당 방식으로 수정하여 해결하였다.
다만 **잠재적 결함**으로 기록해 둔다.

### 시나리오 3: 통계 조회 폴링

**구성**: 50개 방(각 10명 투표 완료) setup 후, 50개 방 중 랜덤 선택하여 통계 API 반복 호출 (sleep 0.5s).
Caffeine 캐시 TTL=500ms이므로, 캐시 키당 초당 최대 2회 미스 발생.
50개 캐시 키 x 초당 2회 = **초당 최대 100건의 DB 쿼리**가 발생할 수 있다.

#### 안정 구간 (150 VU)

p50=43ms, p95=224ms. 쓰기 시나리오 대비 **5배 높은 VU**에서도 p50이 비슷한 수준이다.
이유는 두 가지이다:

1. **캐시 히트율이 높다**: 150 VU x 초당 ~2회 = 초당 ~300 요청 중, 캐시 미스는 50키 x 2회 = 100건. 나머지 200건은 DB 접근 없이 캐시에서 즉시 응답한다.
2. **읽기 전용이다**: DB에 접근해도 SELECT만 실행하므로 커넥션 점유 시간이 짧고, 쓰기 잠금이 없어 경합이 발생하지 않는다.

p95(224ms)와 p99(419~580ms)의 큰 차이는, **캐시 미스 시 DB 쿼리**(참여자 조회 + 투표 조회 + 통계 집계)와 **GC STW**(max 1.5s)가 겹치는 최악의 케이스로 추정된다.

#### 한계 구간 (500 VU)

| 지표 | 150 VU | 500 VU | 증가율 |
|------|--------|--------|--------|
| p50 | 43ms | 1.01s | **23.5x** |
| p95 | 224ms | 2.29s | **10.2x** |
| p99 | 419ms | 2.81s | **6.7x** |
| RPS | 110/s | 173/s | 1.6x |

p50이 43ms→1.01s로 **23배 폭증**하였다. 이 시점의 병목은 Hikari가 아닌 **Tomcat 스레드 풀(15개)**이다:

- 500 VU가 동시 요청 → 15개 스레드만 처리 가능 → 485개가 대기
- 캐시 히트로 DB 부하는 낮지만, **스레드 배정 자체를 못 받아** 대기
- RPS가 110→173으로 57% 증가에 그친 것이 이를 증명: 스레드가 포화되면 VU를 아무리 올려도 처리량이 선형 증가하지 않음

p99의 증가율(6.7x)이 p50(23.5x)보다 작은 점이 흥미롭다.
안정 구간에서 이미 p99가 419ms로 높았기 때문에(캐시 미스 + GC), 한계 구간에서의 **상대적** 증가폭은 작다.
반면 p50은 캐시 히트로 43ms였던 것이 **스레드 대기 시간**이 더해지면서 폭등한 것이다.

**성능 경계: 150~300 VU (RPS 110~173) 사이에서 p50이 100ms를 넘는다.**

#### 극한 구간 (4차, ~733 VU — 에러 발생으로 자동 중단)

| 지표 | 500 VU | ~733 VU | 증가율 |
|------|--------|---------|--------|
| p50 | 1.01s | 2.22s | **2.2x** |
| p95 | 2.29s | 3.16s | **1.4x** |
| p99 | 2.81s | 3.34s | **1.2x** |
| RPS | 173/s | 151/s | **0.87x (감소)** |

**~733 VU에서 `connection reset by peer` 에러 4건이 동시 발생**하며 테스트가 자동 중단되었다.
전체 부하테스트 중 **유일하게 에러가 발생한 지점**이다.

에러 원인: 서버(또는 OS 레벨)가 TCP 연결을 강제로 끊음(RST). Tomcat max-threads(15) + accept-count(5) = 동시 수용 20개를 초과한 나머지 연결이 OS TCP 백로그에서 거부된 것으로 추정.

RPS가 173→151로 **오히려 감소**한 것은, 과부하로 서버의 전반적인 처리 효율이 저하되었기 때문이다.
스레드가 포화된 상태에서 VU를 더 올리면 처리량이 늘지 않고 대기 시간만 증가하며, 일정 수준을 넘으면 시스템 리소스(메모리, GC, 컨텍스트 스위칭)가 추가로 소모되어 처리량이 오히려 줄어든다.

## 4. 병목 구조 분석

```
클라이언트 요청
    ↓
[Nginx] ← 레이트리밋 (테스트 중 비활성화)
    ↓
[Tomcat 스레드 풀: 15개] ← 읽기 작업의 1차 병목
    ↓
[Caffeine 캐시] ← 히트 시 DB 우회 (통계 조회)
    ↓
[Hikari 커넥션 풀: 5개] ← 쓰기 작업의 1차 병목
    ↓
[DB]
```

### 쓰기 작업의 병목 체인

1. Tomcat 스레드를 확보 (15개 중 1개)
2. `@Transactional` 진입 시 Hikari 커넥션 확보 시도 (5개 중 1개)
3. **커넥션 풀 포화 시 대기** ← 여기서 지연 발생
4. 커넥션 확보 후 쿼리 실행 (INSERT 또는 SELECT+DELETE+INSERT)
5. 트랜잭션 커밋, 커넥션 반환

쓰기 작업은 Tomcat 스레드를 확보해도 **Hikari 커넥션을 기다리느라** 스레드를 점유한 채 블로킹된다.
투표 수정처럼 커넥션 점유 시간이 긴 작업일수록 풀 회전율이 떨어져 병목이 심해진다.

### 읽기 작업의 병목 체인

1. Tomcat 스레드 확보 시도 (15개 중 1개)
2. **스레드 풀 포화 시 대기** ← 여기서 지연 발생
3. 캐시 조회 → 히트 시 즉시 응답 반환
4. 캐시 미스 시 Hikari 커넥션 확보 → SELECT → 응답

캐시 덕분에 대부분의 요청이 DB에 접근하지 않으므로 Hikari는 병목이 아니다.
대신 **Tomcat 스레드 15개가 먼저 소진**되어 요청 자체가 처리 대기열에 쌓인다.

### GC STW의 영향

모든 시나리오에서 VU 수와 무관하게 **max가 1~3.5초**로 간헐 발생하였다.
이는 JVM Garbage Collection의 Stop-The-World 구간으로 추정되며, 부하가 높을수록 GC 빈도가 증가하여 p99에 영향을 준다.

## 5. 핵심 발견 요약

### 성능 한계

| 기준 | 쓰기 (방 생성) | 쓰기 (투표) | 읽기 (통계 조회) |
|------|---------------|------------|-----------------|
| p50 < 100ms 유지 가능한 VU | ~60 VU | ~50 VU | ~200 VU |
| p95 < 500ms 유지 가능한 VU | ~100 VU | ~100 VU | ~300 VU |
| p50 < 1s 유지 가능한 VU | ~300 VU | ~150 VU | ~500 VU |
| 에러 발생 VU | 500 VU에서도 0% | 500 VU에서도 0% | **~733 VU** |

### Graceful Degradation

쓰기 작업은 500 VU에서도 에러율 0%를 유지하였다. 서버는 과부하 시 에러를 반환하는 대신 응답 시간이 늘어나는 방식으로 대응한다.
이는 Hikari의 connection-timeout(30s)과 Tomcat의 accept-count(5)가 충분한 대기 여유를 제공하기 때문이다.

읽기 작업은 **~733 VU에서 connection reset by peer** 에러가 발생하며 최초 에러를 기록하였다.
Tomcat 스레드 + accept-count를 초과한 연결이 OS 레벨 TCP 대기열에서 거부된 것으로 추정된다.

### 병목 우선순위

1. **Hikari 커넥션 풀 (5개)** — 쓰기 작업의 1차 병목. 동시 쓰기 50 VU 이상에서 대기 발생.
2. **Tomcat 스레드 풀 (15개)** — 읽기 작업의 1차 병목. 동시 읽기 200 VU 이상에서 대기 발생.
3. **GC STW** — 모든 시나리오에서 간헐적 1~3.5초 지연 유발.

### 잠재적 결함

투표 수정 API에 비관적 잠금이 없어, 동일 참여자에 대한 동시 요청 시 race condition이 발생할 수 있다.
현재 서비스 사용 패턴에서는 발생 가능성이 낮지만, 네트워크 재시도 등에 의해 트리거될 수 있다.
